title: "Format & Sintaks"
description: "Cetak biru struktural untuk menyusun hierarki akun sesuai 'Standar Penamaan Akun' versi 'Less Summer'."
---

## Pola Umum

```
[tipe].[kategori].[subkategori].[spesifik]
```

### Contoh

- `assets.cash.wallet.gopay`
- `liabilities.tax.payable.pph21`
- `income.sales.product.digital`
- `expenses.operational.payroll.bpjs_kesehatan`

### Aturan Wajib

1.  Gunakan format huruf kecil dengan segmen yang dipisahkan oleh titik.
2.  Batasi kedalaman hierarki antara **dua hingga lima segmen**, kecuali ada alasan kuat untuk pengecualian.
3.  Untuk transaksi multi-mata uang, gunakan sufiks yang jelas: `assets.cash.bank.bca.idr`, `assets.cash.bank.bca.usd`.
4.  Untuk struktur gabungan atau multi-entitas, sertakan pengenal entitas: `assets.cash.bank.joint.parent_subsidiary`.

## Awalan Numerik (Opsional)

Awalan numerik dapat membantu menyelaraskan akun dengan kode sistem ERP lama tanpa mengubah makna intinya.

* ```1.assets.cash.bank.bca```
* ```5.expenses.operational.marketing.instagram\_ads```

Jika digunakan, pastikan awalan numerik konsisten di seluruh kategori dan dipisahkan dengan titik dari *namespace* utama.

## Penyelarasan Metadata

Hubungkan setiap struktur akun dengan metadata yang terstruktur agar API, basis data, dan model analitik tetap sinkron.

```json
{
  "path": "liabilities.tax.payable.pph23",
  "psak_code": "2110",
  "ifrs_reference": "IAS 12",
  "tax_article": "pph23",
  "normal_balance": "credit"
}
```

Simpan metadata dalam basis data konfigurasi atau repositori *configuration-as-code* untuk mempermudah validasi dan migrasi di masa depan.
